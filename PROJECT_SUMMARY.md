## 🎨 設計亮點

### 1. 真正的樹狀括號結構
不是單純的 5 欄卡片並排，而是用 CSS flex 比例縮放（`.match-slot { flex:1 }`）讓每輪比賽卡片自動垂直置中對齊前一輪的兩場比賽中點，再用 `::before`/`::after` 偽元素畫出直角轉折連接線，呈現與傳統賽程海報相同的視覺效果。

```
16強      8強      準決賽    決賽     冠軍
[8場] ─┬→[4場] ─┬→[2場] ─┬→[1場] ─→ 冠軍
       │        │        │
（連接線把每一對比賽收斂到下一輪，全程自動對齊，無需手動計算座標）
```

### 2. 自動化的對手更新與清除
- 16 強勝者 → 自動填入 8 強對應位置
- 8 強勝者 → 自動填入準決賽對應位置
- 準決賽勝者 → 自動填入決賽對應位置
- 決賽勝者 → 自動顯示為冠軍
- **改選或取消預測時**，會沿著賽程往後遞迴清除已經失效的結果（`clearDownstream()`），不會留下錯誤殘留

### 3. 直接點擊互動設計（無彈窗）
- 點擊隊伍即選定，即時更新，無需二次確認
- 點擊已選定的隊伍可取消；點另一隊可切換
- 視覺反饋（hover 變色、勝者綠底 + ✓）即時確認操作

### 4. 響應式佈局
```
桌面: [16強][8強][準決賽][決賽][冠軍]  ← 5 欄完整顯示
平板: 欄寬與連接線間距自動縮小
手機: 進一步縮小，可橫向滾動查看完整樹狀圖
```

### 5. 友善列印
獨立的 `print.css`（僅 `media="print"` 生效），固定 A4 直式版面，**沿用與螢幕相同的樹狀結構與連接線**，只是縮小尺寸讓 5 欄能塞進較窄的直式頁面寬度，而非退化成簡單列表。

## 🔧 技術實現

### 核心演算法

#### 進程映射（PROGRESSION）
```javascript
r16_1 勝者 → qf_1 team1
r16_2 勝者 → qf_1 team2
r16_3 勝者 → qf_2 team1
...
qf_1 勝者 → sf_1 team1
qf_2 勝者 → sf_1 team2
qf_3 勝者 → sf_2 team1
qf_4 勝者 → sf_2 team2
sf_1 勝者 → f_1 team1
sf_2 勝者 → f_1 team2
f_1 勝者  → champion
```

#### 預測流程（直接點擊，無彈窗）
1. 使用者點擊比賽卡片中的隊伍
2. `selectWinnerDirect()` 判斷是「新選擇」還是「取消/切換」
3. `savePrediction()` 儲存並呼叫 `updateNextRound()` 推進到下一輪
   - 若下一場的隊伍因此改變且原本已有結果，連帶呼叫 `clearDownstream()` 清除後續失效資料
4. 取消選擇時改呼叫 `cancelPrediction()` → `clearDownstream()` 往後遞迴清空
5. `renderBracket()` 重新渲染所有 5 個階段
6. 每次異動都會呼叫 `saveBracket()` 寫入 LocalStorage

### 數據流
```
User Click (Team)
    ↓
selectWinnerDirect()
    ↓
BracketStorage.savePrediction() / cancelPrediction()
    ↓
updateNextRound() / clearDownstream()  ← bracket-data.js
    ↓
saveBracket() → LocalStorage
    ↓
renderBracket() → 重新渲染 5 個階段
```

## 🎓 開發心得

### 設計決策

#### 1. 為什麼採用樹狀結構？
- **清晰直觀** - 淘汰賽的自然呈現方式
- **視覺連接** - 用連接線清楚追蹤隊伍晉級路徑
- **緊湊空間** - 一屏（或一頁）顯示整個賽制

#### 2. 為什麼只預測勝者？
- **簡化操作** - 用戶只需點擊選擇
- **無需複雜輸入** - 無比分輸入欄
- **聚焦核心** - 淘汰賽的本質是勝負

#### 3. 為什麼改成直接點擊而非彈窗？
- **減少操作步驟** - 少一次確認，預測更快
- **降低認知負擔** - 所見即所得

#### 4. 為什麼要支援取消/切換並自動清除下游？
- **避免資料不一致** - 改變早期輪次的選擇時，後面輪次的舊結果如果不清除會產生「幽靈勝者」
- **讓使用者敢於修改預測** - 不用擔心改一場比賽會弄亂整個賽程
